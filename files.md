### `consentflow/sdk.py`
**Reusable consent-check SDK** — the central read path for all gate modules.

**Public API:**
- `is_user_consented(user_id, purpose, *, redis_client, db_pool) -> bool` *(async)*
- `is_user_consented_sync(user_id, purpose, ...) -> bool` *(sync wrapper via `asyncio.run`)*

**Resolution order:**
1. Redis cache — key `consent:{user_id}:{purpose}` (sub-millisecond)
2. PostgreSQL `consent_records` table (authoritative)
3. Returns `False` (deny-by-default) when no record exists

**Connection management:** If `redis_client` or `db_pool` are not supplied, the SDK creates short-lived ad-hoc connections and closes them after the call. For production use, pass the shared app-level pool/client.

---

### `consentflow/anonymizer.py`
**PII detection and masking via Microsoft Presidio.**

**Public API:**
- `anonymize_record(record: dict) -> dict`

**Behaviour:**
- Scans all string-valued fields using `AnalyzerEngine` (spaCy `en_core_web_lg` backend)
- Replaces detected entities with `<REDACTED>` via `AnonymizerEngine`
- Recursively processes nested dicts and lists
- Returns a **new dict** — original is never mutated
- Non-string values (int, float, bool, None) pass through untouched

**Detected entities:** PERSON, EMAIL_ADDRESS, PHONE_NUMBER, CREDIT_CARD, IBAN_CODE, IP_ADDRESS, LOCATION, DATE_TIME, NRP, MEDICAL_LICENSE, URL, and more.

**Performance:** `AnalyzerEngine` and `AnonymizerEngine` are module-level singletons — loaded once on first import and reused for all calls.

### `consentflow/dataset_gate.py`
**Consent-aware dataset registration gate (Step 3).**

**Public API:**
- `register_dataset_with_consent_check(dataset, run_id, *, purpose, redis_client, db_pool, mlflow_experiment) -> GateResult` *(async)*

**`GateResult` dataclass fields:**

| Field | Type | Description |
|-------|------|-------------|
| `run_id` | `str` | Pipeline run identifier |
| `total_records` | `int` | Total records processed |
| `consented_count` | `int` | Records passed through unchanged |
| `anonymized_count` | `int` | Records anonymized by Presidio |
| `mlflow_run_id` | `str` | MLflow active run ID |
| `artifact_path` | `str` | Path of cleaned dataset artifact in MLflow |
| `cleaned_dataset` | `list[dict]` | The processed output dataset |

**Flow per record:**
1. Extract `user_id` from record (treats missing as revoked)
2. Call `is_user_consented(user_id, purpose)`
3. Granted → append unchanged; Revoked → anonymize first
4. Log metrics to MLflow: `total_records`, `consented_count`, `anonymized_count`, `anonymized_ratio`
5. Write cleaned dataset as JSON artifact to MLflow at `dataset_gate/<run_id>_cleaned_dataset.json`

### `consentflow/training_gate.py`
**Kafka consumer that quarantines MLflow training runs on consent revocation (Step 5).**

> ⚠️ Runs as a **standalone process** — not started by the FastAPI app lifespan.
> Launch with: `python -m consentflow.training_gate`

**Key classes / functions:**

- **`QuarantineRecord`** (dataclass) — records one quarantine action: `user_id`, `run_id`, `experiment_id`, `flagged_at`, `reason`, `kafka_offset`, `kafka_partition`
- **`TrainingGateConsumer`** — async Kafka consumer class with injectable `search_runs_fn` and `quarantine_fn` for testing
- **`run_training_gate_consumer()`** — factory that creates a real `AIOKafkaConsumer` and runs the gate loop
- **`TrainingGateConsumer.run()`** — main loop; calls `_process_revocation` per message; handles `asyncio.CancelledError` for clean shutdown

**Flow per Kafka message:**
1. Parse event JSON; extract `user_id`
2. Call `search_runs_by_user(user_id)` via MLflow
3. For each matching run: `apply_quarantine_tags(run_id, user_id, reason, timestamp)`
4. Append a `QuarantineRecord` to `self.quarantine_log`

**Kafka config:** group ID `consentflow-training-gate`, auto offset reset `earliest`, auto commit enabled, raw bytes deserialization.

### `consentflow/inference_gate.py`
**ASGI middleware for inference-time consent enforcement (Step 4).**

**Class:** `ConsentMiddleware(BaseHTTPMiddleware)`

**Constructor params:**
- `protected_prefixes: list[str]` — paths to guard (default: `["/infer"]`)
- `purpose: str` — consent purpose string (default: `"inference"`)

**User-ID extraction order:**
1. `X-User-ID` HTTP header (fast, no body read required)
2. `user_id` field in JSON body (POST/PUT/PATCH only)

**Response behaviour:**
| Condition | HTTP Status |
|-----------|-------------|
| No `user_id` found | `400 Bad Request` |
| Consent revoked | `403 Forbidden` |
| Consent service error | `503 Service Unavailable` (fail-closed) |
| Consent granted | Request forwarded to handler |

**Reuses** shared `redis_client` and `db_pool` from `app.state` when available; creates ad-hoc connections otherwise (e.g. in tests).

### `consentflow/langchain_gate.py`
**LangChain callback handler for consent enforcement (Step 4 — LLM variant).**

**Class:** `ConsentCallbackHandler(BaseCallbackHandler)`

**Constructor params:**
- `user_id: str | UUID`
- `purpose: str` (default: `"inference"`)
- `redis_client`, `db_pool` — optional shared connections

**Hooks implemented:**
- `on_llm_start()` — synchronous; invoked before every LLM call in sync chains
- `on_llm_start_async()` — async; invoked in async LCEL pipelines

**Exception:** `ConsentRevokedException(RuntimeError)` — raised with `user_id` and `purpose` attributes when consent is revoked. Callers should catch this to return structured errors.

### `consentflow/monitoring_gate.py`
**Consent-aware Evidently AI drift monitor (Step 6).**

**Key classes:**

- **`DriftAlert`** (dataclass) — one alert per revoked user in a drift window: `user_id`, `window_start`, `window_end`, `revoked_count`, `severity`, `timestamp`
- **`DriftCheckResult`** (dataclass) — full result: `tagged_df`, `report`, `alerts`, `has_revoked_samples`, `revoked_count`
- **`ConsentAwareDriftMonitor`** — orchestrator class

**Constructor params:**
- `consent_fn: Callable[[str, str], bool]` — defaults to `is_user_consented_sync`; inject a dict-lookup in tests
- `purpose: str` — consent purpose (default: `"monitoring"`)
- `severity_threshold: int` — rows at/above which severity becomes `"critical"` (default: `5`)

**Key methods:**

| Method | Description |
|--------|-------------|
| `tag_samples_with_consent(df, user_id_col)` | Adds `_consent_status` column to DataFrame |
| `run_drift_report(reference_df, current_df)` | Runs Evidently `DataDriftPreset` (lazy import) |
| `check_for_revoked_samples(tagged_df, ...)` | Emits `DriftAlert` per unique revoked user |
| `run_consent_aware_drift_check(...)` | Orchestrates all 3 steps above |

**Severity logic:** `"warning"` when revoked row count < threshold; `"critical"` when ≥ threshold.

Evidently is imported lazily (inside `run_drift_report`) so the module can be used without Evidently installed if `run_evidently=False` is passed.

**OpenTelemetry tracer factory (Step 7).**

**Functions:**

**Function:** `traced_register_dataset(dataset, run_id, *, tracer, db_pool, redis_client, purpose, mlflow_experiment) -> GateResult` *(async)*

### `consentflow/app/routers/consent.py`
**Consent CRUD endpoints. Router prefix: `/consent`**

| Endpoint | Description |
|----------|-------------|
| `POST /consent` | Upsert record; invalidates Redis cache on every write. `ON CONFLICT (user_id, purpose, data_type) DO UPDATE`. Returns `ConsentRecord`. |
| `POST /consent/revoke` | Sets `status='revoked'` for all data_type rows under `user_id + purpose`; invalidates cache. Returns the last updated `ConsentRecord`. |
| `GET /consent/{user_id}/{purpose}` | Redis cache first → PostgreSQL fallback; populates cache on DB hit. Returns `ConsentStatusResponse` with `cached: bool`. |

### `consentflow/app/routers/webhook.py`
**Webhook ingress for external consent-revocation signals. Router prefix: `/webhook`**

**`POST /webhook/consent-revoke`** — Accepts `OneTrustRevokePayload` (camelCase fields):
- `userId`, `purpose`, `consentStatus` (must be `"revoked"`), `timestamp`

**Idempotent** — uses `INSERT ... ON CONFLICT` with `data_type='webhook'` when no prior record exists.

**Flow:**
1. Validate `consentStatus == "revoked"` (422 otherwise)
2. Parse `userId` as UUID (422 if invalid)
3. Upsert revocation to PostgreSQL
4. Invalidate Redis cache key
5. Publish `consent.revoked` event to Kafka

**Responses:** `200` (fully propagated) / `207` (DB+cache OK, Kafka failed) / `422` / `500`

## `consentflow/migrations/`

### `consentflow/migrations/001_init.sql`
**Initial database schema.** Applied automatically at app startup.

Creates:
- Extension: `pgcrypto` (for `gen_random_uuid()` on older Postgres)
- Table `users`: `id UUID PK`, `email TEXT UNIQUE`, `created_at TIMESTAMPTZ`
- Table `consent_records`: `id UUID PK`, `user_id UUID FK→users(CASCADE)`, `data_type TEXT`, `purpose TEXT`, `status TEXT CHECK('granted','revoked')`, `updated_at TIMESTAMPTZ`
- Unique index: `(user_id, purpose, data_type)` — enables upsert
- Query index: `(user_id, purpose, status)` — covers hot lookup pattern

### `consentflow/migrations/002_audit_log.sql`
**Audit log schema.** Applied automatically at app startup after `001_init.sql`.

Creates:
- Table `audit_log`: `id UUID PK`, `event_time TIMESTAMPTZ`, `user_id TEXT` (not UUID — supports aggregate contexts), `gate_name TEXT`, `action_taken TEXT`, `consent_status TEXT`, `purpose TEXT (nullable)`, `metadata JSONB (nullable)`, `trace_id TEXT (nullable)`
- Indexes: `idx_audit_log_user_id`, `idx_audit_log_event_time DESC`, `idx_audit_log_gate_name`

Valid `action_taken` values: `passed` | `blocked` | `quarantined` | `alerted` | `anonymized`
